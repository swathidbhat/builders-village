# Bug Log

Track bugs encountered during development. Newest at the top.

| Date | Expected Behavior | Actual Behavior | Root Cause | Fix |
|------|-------------------|-----------------|------------|-----|
| 2026-03-11 | Village stays visible after closing interior view | Entire village disappears (blank canvas, only background color + header visible) after closing the interior modal | InteriorView creates its own `PIXI.Application` and calls `app.destroy(true)` on unmount. In PixiJS v8, destroying a renderer can invalidate shared GPU resources (texture pools, buffer caches) that the village's renderer also depends on. The village app stays alive but its GPU-backed textures (ground, buildings, trees, characters) become invalid. | Replaced `app.destroy(true)` with manual cleanup: stop ticker, remove canvas from DOM, destroy stage children. Avoids calling `renderer.destroy()` entirely so shared GPU state is not corrupted. |
| 2026-03-10 | Text readable on dark backgrounds; carried box visible on dark village ground | Text appeared dim/muddy (brown-on-brown); box invisible against ground | **Systemic / perceptual**: The `village` Tailwind palette is entirely warm brown tones. While shades like `village-200` (`#d4c4aa`) technically pass WCAG AA math (~10.5:1 vs `village-900`), **brown text on brown backgrounds is perceptually muddy** — same-hue family blurs together regardless of luminance. White provides 2–3x better visual contrast. Repeatedly stepping `text-village-400` → `300` → `200` never fixed readability because the problem was hue sameness, not luminance. Separately, the carried box was set to `0x3a2215` (near-black brown) making it invisible on dark ground. | **Rule**: Never use `text-village-*` for text. Use `text-white` / `text-white/70` / `text-white/60` only. Village palette is for backgrounds and borders. Added `contrastRatio()` + `relativeLuminance()` to `colors.ts` and automated tests in `ui-contrast.test.ts`: (1) static scan rejecting any `text-village-*` in `.tsx` files, (2) assertions that white provides 2–3x better contrast than village shades, (3) WCAG assertions on all chosen text/bg pairs, (4) box visibility checks. |
| 2026-03-10 | Buildings render on initial load | Only ground tiles visible, no buildings | Race condition: WebSocket state arrived before PixiJS async `app.init()` finished. The `useEffect([state])` fired while `sceneRef` was null. After init completed, it used the stale empty-state closure. | Store latest state in a ref (`pendingStateRef`). After `app.init()` completes, apply `pendingStateRef.current`. |
| 2026-03-10 | Project name "fallacy-chrome-extension" | Got "swathibhat-Documents-GitHub-fallacy-chrome-extension" | `extractProjectName()` stopped at the first gap between known prefixes (the username segment). It should find the *last* known prefix. | Rewrote algorithm to scan all segments and take the slice after the last known prefix. |
| 2026-03-10 | PixiJS Graphics API draws polygon shapes | Buildings invisible despite no errors | In PixiJS v8, the API is shape-first-then-fill: `g.poly([...]).fill(color)`. Original code called `g.fill(color)` before defining the path (v7 pattern), then `g.fill()` after. | Rewrote all sprite generators to use v8 `g.poly([...]).fill(color)` / `g.rect(...).fill(color)` pattern. |
| 2026-03-10 | Claude Code catch-all "Documents" directory filtered out | Shows as a project named "Documents" | Claude Code creates a `-Users-swathibhat-Documents` project dir for sessions not tied to a specific project. Our watcher treats it as a project. | Filter out projects whose name resolves to a known system directory. |
